แนวคิดสำคัญ

Key Concepts ที่ต้องเข้าใจ

1. วิวัฒนาการของข้อมูล (Data Evolution)▼

ข้อมูลมีการพัฒนาตามลำดับขั้นตอน เริ่มจากสิ่งที่เป็นวัตถุดิบไปจนถึงซอฟต์แวร์ที่ใช้จัดการ:

📦
Data Object

ข้อเท็จจริงดิบ (Raw Fact) เช่น นักศึกษา, พนักงาน

↓
📋
Data

มีโครงสร้าง Schema (ID, Name) และ Instance (101, Ake)

↓
🗄️
Database

ฐานข้อมูลรูปแบบตารางและไฟล์ เช่น student.dat, .mdb

↓
⚙️
DBMS

ซอฟต์แวร์จัดการ เช่น MS-SQL, MySQL, Oracle

ข้อควรระวัง

Data Object คือสิ่งที่เราจะเก็บ (เทียบเท่า Entity) เช่น นักศึกษา, สินค้า ส่วน Schema คือโครงสร้าง (เทียบเท่า Attribute) เช่น ID, Name และ Instance คือข้อมูลจริงๆ เช่น 101, Ake

2. Database vs DBMS▼

Database (ฐานข้อมูล)

แหล่งเก็บข้อมูลทางกายภาพ (ไฟล์บนดิสก์) เป็นที่ที่ข้อมูลถูกจัดเก็บจริงๆ เช่น ไฟล์ student.dat, .mdb

DBMS (ระบบบริหารจัดการฐานข้อมูล)

ซอฟต์แวร์ที่ติดตั้งบนเซิร์ฟเวอร์เพื่อเป็นตัวกลางในการเชื่อมต่อ บริหารจัดการ และดูแลความปลอดภัยให้กับข้อมูล เช่น MS-SQL Server, MySQL, Oracle

ข้อผิดพลาดที่พบบ่อย

อย่าสับสนระหว่าง Database กับ DBMS! Database คือตัว "ข้อมูล" ที่ถูกจัดเก็บ แต่ DBMS คือตัว "ซอฟต์แวร์/ระบบ" ที่เข้าไปควบคุมบริหารจัดการ

3. Island of Information▼

ปัญหา Island of Information เกิดจากการที่แต่ละแผนกต่างคนต่างเก็บแฟ้มข้อมูลของตนเอง (เช่น ใช้ไฟล์ Excel แยกกัน) ทำให้ข้อมูลไม่เชื่อมโยงกันและเกิดความซ้ำซ้อน

วิธีแก้ไข

การใช้ระบบฐานข้อมูลรวมศูนย์ (Centralized Database) ช่วยแก้ปัญหานี้ได้ โดยเชื่อมโยงข้อมูลทั้งหมดเข้าไว้ในที่เดียว

ปัญหาที่ตามมา

  • Data Inconsistency: ข้อมูลไม่สอดคล้องกัน เช่น การสะกดชื่อต่างกัน หรือการใช้คำย่อกับคำเต็มปะปนกัน
  • Data Redundancy: ข้อมูลซ้ำซ้อน ทำให้สิ้นเปลืองพื้นที่จัดเก็บ
4. Dense Data vs Sparse Data▼

การออกแบบตารางที่ดีควรให้ได้ข้อมูลที่มีความหนาแน่น เพื่อประหยัดพื้นที่จัดเก็บบนเซิร์ฟเวอร์

Sparse Data (ข้อมูลเบาบาง)

มีช่องว่างมากเกินไปในตาราง ทำให้สูญเสียพื้นที่บนเซิร์ฟเวอร์โดยเปล่าประโยชน์ เกิดจากการนำ Data Object หลายเรื่องมารวมในตารางเดียวกัน

Dense Data (ข้อมูลหนาแน่น)

มีข้อมูลครบทุกช่อง เป็นเป้าหมายสูงสุดของการออกแบบตาราง ทำได้โดยการแยก Object ต่าง类型กันออกเป็นหลายๆ ตาราง

วิธีแก้ Sparse Data

ต้องวิเคราะห์และแยก Object ออกจากกัน เช่น ตารางพนักงาน ตารางทักษะ ตารางประกาศนียบัตร แยกกันคนละตาราง เพื่อให้ได้ Dense Data

5. Data Integrity & Inconsistency▼

ฐานข้อมูลที่ดีต้องมีความสมบูรณ์ถูกต้องของข้อมูล (Integrity) ตลอดเวลา และต้องป้องกันปัญหาข้อมูลไม่สอดคล้องกัน (Inconsistency)

ตัวอย่าง Inconsistency

  • มีคนพิมพ์ชื่อ Peter พลาดเป็น Pter
  • พิมพ์ตัวย่อ DB กับ Database สลับกันในฐานข้อมูลเดียวกัน

วิธีป้องกัน

บังคับใช้โครงสร้าง Schema ที่ชัดเจนและอ้างอิงจาก Data Dictionary (พจนานุกรมข้อมูล) เพื่อให้ทุกคนใช้คำศัพท์เดียวกัน

Database Careers

1. Database Administrator (DBA)▼

ผู้ดูแลระบบและติดตั้งซอฟต์แวร์ DBMS เป็นสายงานที่มีเงินเดือนสูงสุด (ประมาณ 50,000 บาทขึ้นไปสำหรับเริ่มต้น)

หน้าที่หลัก

  • ดูแลภาพรวมของระบบฐานข้อมูลทั้งหมด
  • ดูแลความปลอดภัยของข้อมูล
  • ดูแลรักษาเซิร์ฟเวอร์
  • ติดตั้งและดูแลซอฟต์แวร์ DBMS
2. Database Designer▼

นักออกแบบฐานข้อมูล ทำหน้าที่วิเคราะห์ว่าในธุรกิจมี Data Object อะไรบ้าง และเขียนโครงสร้างออกมา มีรายได้รองลงมา (ประมาณ 30,000 บาท)

หน้าที่หลัก

  • วิเคราะห์ Data Object ทั้งหมดในองค์กร
  • ออกแบบโครงสร้างตาราง (Schema)
  • กำหนดความสัมพันธ์ระหว่างตาราง
3. Database Developer▼

นักพัฒนาผู้เขียนโปรแกรม (เช่น สร้างเว็บไซต์หรือ UI) ดึงข้อมูลไปใช้งาน เป็นผู้มีความรู้ระดับโครงสร้างตารางทั่วไป รายได้เริ่มต้นต่ำสุดในสามกลุ่ม (ประมาณ 20,000 บาท)

หน้าที่หลัก

  • เขียนโปรแกรมเชื่อมต่อกับฐานข้อมูล
  • สร้าง UI สำหรับแสดงข้อมูล
  • ดึงข้อมูลจาก DB มาใช้งาน

เปรียบเทียบรายได้

5 Components of Database Environment

ระบบฐานข้อมูลมีองค์ประกอบ 5 ประการหลักที่ทำงานร่วมกัน หากข้อสอบถาม ต้องตอบให้ครบถ้วน:

1. Hardware▼

อุปกรณ์ทางกายภาพ เช่น เครื่องเซิร์ฟเวอร์ ฮาร์ดดิสก์ หน่วยความจำ RAM ที่ใช้ในการเก็บและประมวลผลข้อมูล

2. Software▼

ตัวโปรแกรม DBMS ซอฟต์แวร์ระบบปฏิบัติการ และซอฟต์แวร์-application ที่ใช้เชื่อมต่อกับฐานข้อมูล

3. People▼

ผู้ที่เกี่ยวข้องทั้งหมด เช่น DBA, Developer, Designer และ User ผู้ใช้งานทั่วไป

4. Procedure/Function▼

ขั้นตอนการทำงานและหน้าที่ต่างๆ ภายในระบบ รวมถึงกฎเกณฑ์การใช้งาน

5. Data & Business Logic▼

ตัวข้อเท็จจริงที่เก็บรวบรวมไว้ รวมถึง "เงื่อนไขทางธุรกิจ" ซึ่งจะแตกต่างกันไปตามบริบทของแต่ละองค์กร

หน้าที่หลักของ DBMS

DBMS Functions

1. Data Dictionary Management▼

การจัดการพจนานุกรมข้อมูลเพื่อกำหนดความหมายของข้อมูลแต่ละตัว ป้องกันความสับสนเรื่องคำศัพท์

2. Data Storage Management▼

บริหารพื้นที่การจัดเก็บข้อมูลบนฮาร์ดดิสก์ จัดการโครงสร้างไฟล์และดัชนี

3. Data Transformation▼

การแปลงรูปแบบข้อมูล เช่น ในตารางเก็บปีแบบคริสตศักราช แต่เวลาแสดงบนเว็บไซต์ดึงข้อมูลด้วยคำสั่ง SQL แปลงให้แสดงเป็นพุทธศักราชและแสดงชื่อเดือนแบบเต็ม

ตัวอย่าง Data Transformation ด้วย SQL
-- แปลงวันเกิดจากคริสตศักราชเป็นพุทธศักราช
SELECT name,
       CONCAT(
         YEAR(birthdate) + 543,
         ' ',
         MONTHNAME(birthdate),
         ' ',
         DAY(birthdate)
       ) AS buddhist_date
FROM student;

ทำไมใช้ SQL ไม่ใช้ Python หรือ C?

เพราะสามารถให้ฝั่งหลังบ้าน Database จัดการได้เลยผ่านคำสั่ง SQL ทำให้รวดเร็วและเป็นศูนย์กลาง ไม่ต้องพึ่งพาภาษาโปรแกรมหน้าบ้าน

4. Security Management▼

การจัดการความปลอดภัย เพื่อป้องกันไม่ให้ข้อมูลสูญหายหรือถูกดัดแปลงง่ายๆ เหมือนที่เกิดกับการใช้ไฟล์ Excel ธรรมดา

5. Multiuser Access Control▼

การดูแลให้ผู้ใช้หลายคน (Multiuser) สามารถใช้งานพร้อมกันได้โดยระบบไม่พัง จัดการเรื่อง Concurrent Access

6. Backup & Recovery▼

สามารถกู้คืนข้อมูลให้กลับสู่สภาวะปกติ (Recover) ได้หากเซิร์ฟเวอร์มีปัญหาหรือแครช (Crash)

7. Data Integrity Management▼

คงสภาพความสมบูรณ์และถูกต้องของข้อมูล ป้องกันข้อมูลเสียหายหรือไม่สอดคล้องกัน

การออกแบบข้อมูล

Dense vs Sparse Data และ Inconsistency

ข้อผิดพลาดร้ายแรงของการออกแบบฐานข้อมูลคือการนำ Data Object ที่ต่างชนิดกันไปยัดรวมไว้ใน "ตารางเดียวกัน" ซึ่งจะทำให้เกิดปัญหา Sparse Data

ตัวอย่าง: Sparse Data vs Dense Data
-- ❌ Sparse Data: ยัดทุกอย่างในตารางเดียว
--
-- | ID | Name  | Skill     | Cert         |
-- |----|-------|-----------|--------------|
-- | 1  | Ake   | SQL       | NULL         |
-- | 2  | B     | NULL      | AWS Cert     |
-- | 3  | C     | Python    | NULL         |
-- (ช่อง NULL เยอะ = เปลืองพื้นที่)

-- ✅ Dense Data: แยก Object ออกมาเป็นหลายตาราง
--
-- Table: Employee
-- | ID | Name |
-- |----|------|
-- | 1  | Ake  |
--
-- Table: Skill
-- | Emp_ID | Skill |
-- |--------|-------|

โครงสร้างข้อมูลและการใช้ XML

Data Formats & XML

1. ประเภทของรูปแบบข้อมูล▼

Structural Data (ข้อมูลแบบมีโครงสร้าง)

เช่น ข้อมูลในรูปแบบตาราง Table ของ DBMS หรือ Spreadsheet มีการกำหนดคอลัมน์และข้อจำกัดชัดเจน ไม่สามารถสุ่มใส่ข้อมูลประเภทต่างกันได้

Unstructural Data (ข้อมูลแบบไม่มีโครงสร้าง)

เช่น ข้อความ (Text Format) เป็นประโยคภาษาอังกฤษที่ต้องใช้โปรแกรมตัดคำอีกทีเพื่อหาข้อมูล ไม่มีรูปแบบตายตัว

Semi-structural Data (ข้อมูลกึ่งโครงสร้าง)

เช่น ภาษามาร์กอัป (Markup Language) อย่าง HTML หรือ XML (Extensible Markup Language) มีแท็กกำกับแต่ไม่เข้มงวดเท่า Structural

2. การส่งข้อมูลข้ามระบบด้วย XML▼

เมื่อมีการเชื่อมโยงข้อมูลระหว่างสององค์กร เช่น มหาวิทยาลัยมหิดล (ICT) และธนาคารไทยพาณิชย์ (SCB) ทั้งสองแห่งอาจใช้ซอฟต์แวร์ DBMS ที่ต่างกันโดยสิ้นเชิง

ปัญหา

  • มหาวิทยาลัยใช้ MS-SQL Server ส่วนธนาคารใช้ Oracle
  • โครงสร้างไฟล์ไม่ตรงกัน: ฝั่งมหาวิทยาลัยใช้ student.dat ส่วนธนาคารใช้ customer.dat
  • รูปแบบคำสั่ง Syntax ไม่เหมือนกัน

ทางแก้: ใช้ XML เป็นภาษากลาง (Standard Format)

สององค์กรตกลงรูปแบบมาตรฐานร่วมกันโดยใช้ XML (Semi-structural format) เพื่อเป็น "ภาษากลาง" ในการแลกเปลี่ยนข้อมูล

ตัวอย่าง XML ที่ส่งข้ามระบบ
<Student>
  <id>101</id>
  <name>Ake</name>
  <faculty>ICT</faculty>
  <tuition_fee>15000</tuition_fee>
</Student>

XML vs HTML

  • HTML: มีแท็กมาตรฐานถูกกำหนดโดยหน่วยงาน W3C ใช้แสดงผลผ่าน Browser
  • XML: เป็นแท็กที่ 2 องค์กรสามารถกำหนดข้อตกลง (Structure) ร่วมกันเองได้ เพื่อส่งข้อมูลเฉพาะกิจ

Exam Notes

เคล็ดลับการเก็งข้อสอบ

1. Database vs DBMS

จุดที่มักออกสอบ: ความแตกต่างระหว่าง Database กับ DBMS Database = แหล่งเก็บข้อมูลทางกายภาพ | DBMS = ซอฟต์แวร์จัดการ

2. Sparse Data

สถานการณ์ประยุกต์: หากให้ตารางมาแล้วช่องว่าง (Null) เยอะเกินไป เรียกว่า Sparse Data แก้ไขโดยแตก Data Object ให้กลายเป็นหลายตารางแยกกัน

3. XML สำหรับเชื่อมต่อข้ามองค์กร

การเชื่อมต่อข้ามองค์กร: หากองค์กร A ใช้ SQL Server องค์กร B ใช้ Oracle จะแลกเปลี่ยนข้อมูลโดยใช้ XML เป็นตัวกลาง Standard Format

4. Data Transformation ด้วย SQL

แม้จะเป็นการแสดงผลหน้าเว็บไซต์ แต่ทำไมเราไม่ใช้ C หรือ Python แปลงข้อมูล? เพราะสามารถให้ฝั่งหลังบ้าน Database จัดการได้เลยผ่านคำสั่ง SQL

สรุปสิ่งที่ต้องจำ (Exam Focus!)

  • องค์ประกอบ 5 อย่าง: Hardware, Software, People, Procedure, Data/Business Logic
  • เงินเดือน: DBA สูงกว่า Designer สูงกว่า Developer
  • DBMS Functions: Data Dictionary, Transformation, Access Control, Security, Backup & Recovery
  • Dense Data: มุ่งเป้าไปที่ความหนาแน่น ลด Sparse Data โดยแยกตารางตาม Data Object
  • XML: Semi-structural ใช้เป็นตัวกลางเชื่อมต่อและส่งต่อข้อมูลระหว่างองค์กรที่ใช้ DBMS ต่างกัน

  • XML ไม่ใช่ HTML: HTML แสดงผลผ่าน Browser แท็ก由 W3C กำหนด | XML ส่งข้อมูลข้ามระบบ แท็กกำหนดเองได้
