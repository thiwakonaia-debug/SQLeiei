บทนี้เรียนเรื่องอะไร?

วัตถุประสงค์การเรียนรู้

สิ่งที่นักศึกษาควรเข้าใจอย่างถ่องแท้หลังเรียนจบบทนี้:

  • สามารถแยกแยะความแตกต่างระหว่าง Data, Information, Knowledge และ Wisdom ได้
  • เข้าใจปัญหาและข้อจำกัดของระบบแฟ้มข้อมูล (File Systems) โดยเฉพาะปัญหาข้อมูลซ้ำซ้อนและความผิดปกติของข้อมูล (Data Anomalies)
  • อธิบายองค์ประกอบ 5 ส่วนของระบบฐานข้อมูล และหน้าที่สำคัญของ DBMS ได้
  • เข้าใจหลักการออกแบบฐานข้อมูลที่ดีเพื่อแก้ปัญหา Sparse Data ให้เป็น Dense Data
  • สามารถบอกความแตกต่างของประเภทฐานข้อมูล และบทบาทหน้าที่ของสายอาชีพ Database (DBA, Designer, Developer) พร้อมโครงสร้างรายได้

2. ปัญหาของระบบแฟ้มข้อมูล (File Systems Issues)▼

เมื่อก่อนเราใช้ Excel หรือแฟ้มแยกกันในแต่ละแผนก ทำให้เกิดปัญหา:

  • Island of Information: ข้อมูลถูกเก็บแยกจากกันเป็นเกาะๆ เช่น ฝ่ายขายเก็บไฟล์นึง ฝ่ายบุคคลเก็บอีกไฟล์ เวลาจะดึงข้อมูลมาทำรายงานรวมต้องสร้างไฟล์ชั่วคราวให้วุ่นวาย
  • Data/Structural Dependence: โครงสร้างข้อมูลผูกติดกับโปรแกรม หากเปลี่ยนขนาดฟิลด์ (เช่น จาก 20 เป็น 30 ตัวอักษร) ต้องแก้โค้ดโปรแกรมใหม่ทั้งหมด
  • Data Redundancy: ข้อมูลเดียวกันถูกเก็บหลายที่ ทำให้กินพื้นที่และเสี่ยงต่อการผิดพลาด
  • Data Anomalies: ผลลัพธ์จากข้อมูลซ้ำซ้อน ทำให้เกิดความผิดปกติ 3 แบบ

Data Anomalies 3 แบบ

  • Insertion anomalies: ความผิดปกติเมื่อเพิ่มข้อมูล
  • Modification/Update anomalies: ความผิดปกติเมื่อแก้ไขข้อมูล
  • Deletion anomalies: ความผิดปกติเมื่อลบข้อมูล

4. Sparse Data vs Dense Data (การออกแบบฐานข้อมูล)▼

Sparse Data (การออกแบบที่แย่)

มักจับวัตถุ (Data Object) หลายชนิดมายัดรวมในตารางเดียว ทำให้เกิดช่องว่าง (Blank) มหาศาลในตาราง สิ้นเปลืองพื้นที่ Harddisk

Dense Data (การออกแบบที่ดี)

ต้องแยก Data Object ออกจากกัน (เช่น แยกตารางพนักงาน, ตารางทักษะ, ตารางใบรับรอง) เพื่อให้ข้อมูลในตารางมีความหนาแน่น ไม่เสียพื้นที่

วิธีแก้

"ต้องแยก Data object ที่ต่างชนิดกัน ออกเป็นคนละตาราง" เช่น ตารางพนักงาน ก็แยกกับ ตารางทักษะความสามารถ

5. รูปแบบของโครงสร้างข้อมูล (Data Structures)▼

Structured Data

ข้อมูลมีโครงสร้างชัดเจน เช่น ตารางในฐานข้อมูล (Table) หรือไฟล์ CSV

Semi-structured Data

ข้อมูลกึ่งโครงสร้าง เช่น XML หรือ JSON ใช้สำหรับเป็นมาตรฐานกลางในการแลกเปลี่ยนข้อมูลระหว่างองค์กรที่มีระบบฐานข้อมูลต่างกัน

Unstructured Data

ข้อความยาวๆ, รูปภาพ, วิดีโอ, เสียง ไม่มีโครงสร้างชัดเจน

เนื้อหาแบบละเอียด

Detailed Explanation

1. ทำไมต้องมีฐานข้อมูล?▼

ทุกกิจกรรมในชีวิตประจำวันล้วนสร้างข้อมูล (Ubiquitous) เช่น การซื้อของออนไลน์, หมอพร้อม, หรือจองตั๋ว องค์กรจึงต้องมีระบบเก็บ รวบรวม ค้นหา และวิเคราะห์ข้อมูลเหล่านั้นเพื่อใช้งาน

2. วิวัฒนาการจาก File Systems สู่ Database Systems▼

อดีตเราใช้ตู้เอกสาร (Manual) พัฒนามาเป็นคอมพิวเตอร์ เช่น Spreadsheet (Excel) ซึ่งโปรแกรมตารางคำนวณเหล่านี้เหมาะกับงานธุรกิจส่วนบุคคล (End-User) แต่ไม่เหมาะกับระบบใหญ่

  • ก่อให้เกิด Island of Information ข้อมูลกระจายตามแผนก
  • ทำให้เกิดความซ้ำซ้อน (Data Redundancy)
  • พอซ้ำซ้อนก็เกิดความผิดปกติเวลา เพิ่ม ลบ หรือแก้ไขข้อมูล (Data Anomalies)

ระบบฐานข้อมูล (Database Systems) จึงเข้ามาแก้ปัญหานี้ โดยดึงข้อมูลมารวมไว้ที่ศูนย์กลาง (Server) และใช้ DBMS จัดการ

3. หน้าที่หลักของ DBMS (DBMS Functions)▼

DBMS ไม่ใช่แค่ที่เก็บข้อมูล แต่มีหน้าที่สำคัญ ได้แก่:

  • จัดการพจนานุกรมข้อมูล (Data Dictionary): ป้องกันปัญหาข้อมูลไม่สอดคล้อง (Data Inconsistency) เช่น พิมพ์ "DB" กับ "Database" หรือ "Peter" กับ "Pter"
  • จัดการการแปลงข้อมูล (Data Transformation): ไม่ต้องเขียนโค้ดภาษา C หรือ Python เพื่อจัด Format ให้เหนื่อย เราสามารถใช้คำสั่ง Database (SQL) แปลงข้อมูล เช่น แปลงปี ค.ศ. 2500 ในระบบหลังบ้าน ให้แสดงเป็น "1 มกราคม พ.ศ. 2500" ที่หน้าจอได้เลย
  • จัดการความปลอดภัย และรองรับการเข้าใช้พร้อมกันหลายคน (Multiuser access control)
  • สำรองข้อมูลและกู้คืนข้อมูล (Backup and recovery)

เรื่องราวที่ช่วยให้เข้าใจ

ตัวอย่าง 1: Business Logic ทางธุรกิจ▼

"Business Logic" คือการกำหนดเงื่อนไขควบคุมฐานข้อมูล ซึ่งจะแตกต่างกันไปในแต่ละองค์กร:

  • มหาวิทยาลัยมหิดล: GPA 2.00 ขึ้นไป ลงทะเบียนได้ 5 วิชา
  • จุฬาฯ: GPA 1.50 ขึ้นไป ลงทะเบียนได้
  • เกษตรศาสตร์: ลงได้ 8 วิชา

กฎเหล่านี้คือ Business logic ที่คนออกแบบระบบต้องนำไปกำหนดเงื่อนไขในฐานข้อมูล

ตัวอย่าง 2: การแลกเปลี่ยนข้อมูลด้วย XML▼

หากคณะ ICT (ใช้ Database ของ SQL Server) ต้องการส่งข้อมูลยอดชำระค่าเทอมไปให้ ธนาคาร SCB (ใช้ Database ของ Oracle) องค์กรสองแห่งนี้ใช้ซอฟต์แวร์คนละค่าย การจะส่งข้อมูลหากันตรงๆ จะทำไม่ได้

จึงต้องอาศัยภาษาตกลงร่วมกันอย่าง XML โดยสร้าง Tag เช่น:

XML Format ตัวอย่าง
<student>
  <id>64010001</id>
  <name>สมชาย ใจดี</name>
  <payment>15,000</payment>
</student>

เพื่อเป็น format มาตรฐานตรงกลางให้เซิร์ฟเวอร์ทั้งสองฝั่งเข้าใจตรงกัน

ตัวอย่าง 3: Data Transformation ด้วย SQL▼

หากเราเก็บข้อมูลในตารางเป็น Data format มาตรฐาน เช่น ปี 2500 แต่ต้องการแสดงผลบนหน้าเว็บไซต์ว่า 1 มกราคม 2500

เราไม่ต้องไปเขียนโค้ดแปลงค่าในภาษาฝั่ง Front-end เราสามารถใช้คำสั่งของ SQL (Database manipulation) แปลง Format ให้ตรงจุดเลยได้

ข้อ 1

ข้อใดคือสาเหตุสำคัญที่องค์กรเปลี่ยนจากระบบ File Systems มาเป็น Database Systems?

A) เพื่อให้เขียนโปรแกรมได้ยาวขึ้น
B) เพื่อแก้ปัญหา Data/Structural Dependence และ Data Redundancy
C) เพื่อลดจำนวน DBA
D) เพราะ Excel คำนวณกราฟไม่ได้แล้ว

เฉลย: B

ระบบเดิมทำให้ข้อมูลผูกติดกับโปรแกรม ถ้าแก้โครงสร้างต้องแก้โค้ดใหม่ และเกิดข้อมูลซ้ำซ้อนจนนำไปสู่ Data Anomalies

เฉลย: B

การนำ Object ที่แตกต่างกันมารวมในตารางเดียว จะทำให้เกิดช่องโหว่ว่างเปล่าจำนวนมาก (Sparse Data) สิ้นเปลืองพื้นที่

Quick Reference - DBMS Chapter 1.2

Data = วัตถุดิบ

Raw facts ยังไม่ผ่านการประมวลผล

Information = ประมวลผลแล้ว

ผ่านการคำนวณ สรุปผล ใช้ตัดสินใจได้

DBMS = ซอฟต์แวร์จัดการ DB

ตัวกลางระหว่างผู้ใช้กับฮาร์ดดิสก์ เช่น MySQL, Oracle, SQL Server

Data Redundancy

ข้อมูลซ้ำซ้อน นำไปสู่ Data Anomalies (ผิดปกติเวลา เพิ่ม/แก้ไข/ลบ)

Business Logic

กฎเกณฑ์เฉพาะของบริษัทที่ใช้คุมข้อมูล

Metadata

Data about Data (ข้อมูลอธิบายคอลัมน์)

Data Dictionary

พจนานุกรมควบคุมคำศัพท์ ป้องกัน Inconsistency

Sparse Data = แย่

ข้อมูลกลวง/โบ๋ เกิดจากการรวม Object หลายชนิดในตารางเดียว

Dense Data = ดี

ข้อมูลอัดแน่นเต็มตาราง เกิดจากการแยก Object ออกเป็นคนละตาราง

XML

ภาษากึ่งโครงสร้าง ใช้สื่อสารระหว่างเซิร์ฟเวอร์คนละยี่ห้อ

DBA = เงินเดือนสูงสุด

ผู้ดูแลระบบ Database ครองตำแหน่งสูงสุดในสายงาน

SQL Basics Cheat Sheet
-- สร้างตาราง
CREATE TABLE table_name (
    id INT PRIMARY KEY,
    name VARCHAR(100)
);

-- เพิ่มข้อมูล
INSERT INTO table_name (id, name) VALUES (1, 'John');

-- ดึงข้อมูล
SELECT * FROM table_name;

-- แก้ไขข้อมูล
UPDATE table_name SET name = 'Jane' WHERE id = 1;

-- ลบข้อมูล
DELETE FROM table_name WHERE id = 1;

Exam Notes

เคล็ดลับการเก็งข้อสอบ

1. องค์ประกอบ 5 อย่างของ DB System

ระวังข้อสอบหลอกเรื่อง "Procedures" ให้จำไว้ว่าในวิชานี้มันคือ "Business Logic" ที่แต่ละองค์กรตั้งกฎไว้ไม่เหมือนกัน (เช่น จุฬา vs มหิดล)

2. XML vs HTML

HTML ใช้โชว์บนบราวเซอร์ (มาตรฐาน W3C) แต่ XML คือการทำ Tag ที่ตกลงกันเอง 2 ฝั่ง ใช้สำหรับเชื่อมโยง Data ระหว่างองค์กร (Semi-structured)

3. ปัญหา Sparse Data

ถ้าโจทย์ถามว่าการรวมข้อมูลพนักงานกับข้อมูลใบประกาศไว้ในตารางเดียวจะเกิดผลเสียอะไร? ตอบ: เกิด Sparse Data เปลืองพื้นที่ วิธีแก้คือต้อง แยกตาราง (Dense Data)

สรุปสิ่งที่ต้องจำ (Exam Focus!)

  • ปัญหา File System: Structural Dependence, Data Dependence, Island of Information, Data Redundancy, Data Anomalies
  • กฎการออกแบบ: ต้องเป็น Dense Data ห้าม Sparse Data วิธีแก้คือแยกตาราง
  • XML: Semi-structured data ใช้แลกเปลี่ยนข้อมูลข้ามองค์กรที่ใช้ซอฟต์แวร์ต่างยี่ห้อ
  • DBA: เงินเดือนสูงสุด เพราะดูแลครบทุกอย่าง
  • Data Dictionary: แก้ปัญหาคำศัพท์ไม่สอดคล้องกัน (Inconsistency)

ข้อควรระวัง - จุดที่มักสับสน

  • Excel ไม่ใช่ DBMS: เป็นแค่โปรแกรม Spreadsheet ระบบแฟ้มข้อมูลประยุกต์ ไม่ใช่ DBMS แบบ MySQL หรือ SQL Server
  • Data Dictionary vs Metadata: Metadata = ข้อมูลอธิบายข้อมูล (เช่น คอลัมน์เป็นตัวเลข 20 หลัก) | Data Dictionary = พจนานุกรมคำศัพท์ขององค์กร ป้องกัน Inconsistency
  • Cloud Computing: ไม่ใช่แค่สถานที่จัดเก็บ แต่เป็นคอนเซปต์การ Share resources/location นำทรัพยากรมาแชร์กัน มีความยืดหยุ่นสูง
