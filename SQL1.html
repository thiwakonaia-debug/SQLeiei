บทนี้เรียนเรื่องอะไร?

บทนี้เป็นจุดเริ่มต้นของการศึกษาวิชา Database Management Systems (DBMS) โดยให้ภาพรวมว่าทำไมเราถึงต้องมีฐานข้อมูล ข้อมูล (Data) แตกต่างจากสารสนเทศ (Information) อย่างไร วิวัฒนาการจากการจัดเก็บด้วยระบบแฟ้มข้อมูล (File Systems) มาสู่ระบบฐานข้อมูล (Database Systems) ข้อดีข้อเสียของการใช้ DBMS ประเภทของฐานข้อมูล รวมถึงสายอาชีพที่เกี่ยวข้องกับการจัดการข้อมูลในปัจจุบัน

วัตถุประสงค์การเรียนรู้

หลังเรียนจบบทนี้ ผู้เรียนควรจะสามารถ:

  • อธิบายความแตกต่างระหว่าง Data, Information, Knowledge และ Wisdom ได้อย่างถูกต้อง

  • สามารถวิเคราะห์และแยกแยะ Data Object รวมถึงกำหนด Primary Key ได้
  • คำนวณพื้นที่จัดเก็บข้อมูลเบื้องต้น (Bits / Bytes) ได้

Key Concepts ที่ต้องเข้าใจ

1. ลำดับขั้นของข้อมูล (Data Hierarchy)▼

ข้อมูลไม่ได้มีเพียงระดับเดียว แต่มีการยกระดับคุณค่าตามการประมวลผล:

  • Data (ข้อมูล): ข้อเท็จจริงดิบ (Raw Facts) ที่เก็บโอนถ่ายมาจากวัตถุ (Data Object)
  • Information (สารสนเทศ): ข้อมูลที่ผ่านการคำนวณ ประมวลผล หรือสรุปผลแล้ว
  • Knowledge (ความรู้): สารสนเทศที่ถูกนำมาวิเคราะห์หาความสัมพันธ์หรือรูปแบบ (Pattern) ทางสถิติ
  • Wisdom (ปัญญา): การตัดสินใจ (Decision Making) ของมนุษย์ที่เกิดจากการนำความรู้มาประยุกต์ใช้

2. Data Object และ Fact▼

เวลาออกแบบฐานข้อมูล เราต้องมองสิ่งต่างๆ เป็น Data Object (วัตถุข้อมูล) เช่น นักศึกษา, สินค้า, ผู้ป่วย

  • วัตถุเหล่านี้จะมี Fact (ข้อเท็จจริง) ติดตัวอยู่เสมอ ซึ่งเป็นสิ่งที่เราเปลี่ยนแปลงไม่ได้ เช่น ชื่อแบรนด์สินค้า, วันเกิด
  • เมื่อเรานำ Fact มาบันทึกลงในสื่อคอมพิวเตอร์ มันจะกลายเป็น Data

ข้อควรระวัง

"อาการไอ" ไม่ใช่คอลัมน์หนึ่งของผู้ป่วย แต่เป็น Object ใหม่ (โรค/อาการ) ที่ต้องแยกสร้างเป็นอีก 1 ตาราง เพราะผู้ป่วยหลายคนแชร์อาการเดียวกันได้

3. ปัญหาของระบบแฟ้มข้อมูล (File System Problems)▼

ก่อนมีฐานข้อมูล เราเก็บข้อมูลแยกเป็นไฟล์ (เช่น Excel แยกระหว่างฝ่ายบุคคลและฝ่ายขาย) ทำให้เกิดปัญหา:

  • Structural Dependence (ยึดติดโครงสร้าง): โปรแกรมต้องรู้โครงสร้างไฟล์ หากเปลี่ยนความยาวตัวอักษร ต้องแก้โค้ดโปรแกรมใหม่
  • Data Dependence (ยึดติดข้อมูล): หากเปลี่ยน Data type (จาก Text เป็น Integer) โปรแกรมพัง ต้องเขียนใหม่
  • Island of Information: ข้อมูลแยกกันอยู่ ทำให้สร้างรายงานรวมยาก
  • Data Redundancy: ข้อมูลซ้ำซ้อน นำไปสู่ Data Anomalies 3 แบบ

Data Anomalies 3 แบบ

  • Insertion Anomaly: เพิ่มข้อมูลไม่ได้
  • Modification Anomaly: แก้ไขแล้วข้อมูลไม่สอดคล้องกัน
  • Deletion Anomaly: ลบข้อมูลแล้วข้อมูลสำคัญอื่นหายไปด้วย
4. โครงสร้างตาราง (Table Structure)▼

เมื่อมองข้อมูลในรูปแบบตาราง:

Field / Attribute (คอลัมน์)

คุณสมบัติของ Object ในแนวตั้ง เช่น รหัสนักศึกษา, ชื่อ, เพศ

Record / Tuple / Row (แถว)

รายการข้อมูล 1 บรรทัด แนวขวาง ที่แสดงข้อมูลของ Object 1 ชิ้น

Key (คีย์)

ข้อมูลที่ใช้จำแนกความแตกต่างของ Object อย่างเฉพาะเจาะจง (มักขีดเส้นใต้ไว้) เช่น รหัสนักศึกษา, บาร์โค้ดสินค้า, ISBN ของหนังสือ

เนื้อหาแบบละเอียด

Detailed Explanation

1. ทำไมต้องมีฐานข้อมูล?▼

ในยุคปัจจุบัน ข้อมูลมีอยู่ทุกหนทุกแห่งและมีปริมาณมหาศาล (Ubiquitous) ไม่ว่าจะเป็นการซื้อของออนไลน์ จองตั๋วเครื่องบิน หรือระบบโรงพยาบาล

เราจึงต้องมี "ฐานข้อมูล" เพื่อจัดเก็บ ค้นหา ประมวลผล และวิเคราะห์ข้อมูลให้เกิดประโยชน์สูงสุด

Key Point

การออกแบบฐานข้อมูลที่ดีช่วยหลีกเลี่ยงความซ้ำซ้อนและได้สารสนเทศที่ถูกต้อง หากออกแบบผิด จะทำให้องค์กรตัดสินใจผิดพลาดได้

2. ความแตกต่างระหว่าง Data และ Information▼

Data (ข้อมูล)

คือข้อเท็จจริง (Raw Fact) ที่ถูกโอนถ่ายมาจากวัตถุลงสู่สื่อ เช่น วันเกิด (Birthdate), น้ำหนัก (Weight), ชื่อ (Name)

Information (สารสนเทศ)

สิ่งที่ได้จากการคำนวณประมวลผล Data เช่น อายุ (Age) เกิดจากการนำปีปัจจุบันลบด้วยปีเกิด หรือ BMI เกิดจากการนำน้ำหนักและส่วนสูงมาคำนวณ

อาจารย์เน้นย้ำ!

เราจะไม่จัดเก็บสารสนเทศ (Information) ไว้ในระบบฐานข้อมูล เพราะมันแปรผัน (Dynamic) และเปลี่ยนแปลงตลอดเวลา (เช่น อายุเปลี่ยนทุกปี) แต่ระบบโปรแกรม (UI) จะดึง Data จากฐานข้อมูลขึ้นมาคำนวณและแสดงผลเป็น Information ให้เราเห็นเอง

4. ระบบแฟ้มข้อมูล (File Systems) และข้อจำกัด▼

ในอดีตเราใช้ระบบแฟ้มข้อมูลแบบทำด้วยมือและพัฒนามาเป็นคอมพิวเตอร์ เช่น การเก็บข้อมูลด้วยโปรแกรม Spreadsheet อย่าง Microsoft Excel

ปัญหาของ File System

  • Structural Dependence: โปรแกรมต้องรู้โครงสร้างไฟล์ เปลี่ยนทีต้องแก้โค้ดทั้งหมด
  • Data Dependence: เปลี่ยน Data type โปรแกรมพังทันที
  • Island of Information: ข้อมูลแยกกันอยู่ สร้างรายงานรวมยาก
  • Data Redundancy: ข้อมูลซ้ำซ้อน → Data Anomalies

Excel ไม่ใช่ Database!

Excel เป็นเพียงโปรแกรมตารางคำนวณ (Computerized file systems) ไม่ได้มีคุณสมบัติของ DBMS ในการจัดการข้อจำกัดโครงสร้างข้อมูลระดับลึก

5. ระบบฐานข้อมูล (Database Systems) และ DBMS▼

องค์ประกอบหลัก

  • Database: คลังเก็บข้อมูลที่มีความสัมพันธ์กัน
  • DBMS: ซอฟต์แวร์ที่ทำหน้าที่จัดการและเป็นตัวกลางระหว่างผู้ใช้กับฐานข้อมูล

ประโยชน์ของ DBMS:

  • ลดความซ้ำซ้อนของข้อมูล
  • ข้อมูลสอดคล้องกัน
  • มีความปลอดภัย
  • ความเป็นอิสระของข้อมูลและโครงสร้าง (Data and Structural Independence)

Database Architecture

ระบบฐานข้อมูลประกอบด้วย 5 ส่วน: Hardware, Software (OS, DBMS, Application), People, Procedures (กฎเกณฑ์การใช้งาน) และ Data

DBMS มีหน้าที่จัดการ:

  • พจนานุกรมข้อมูล (Data Dictionary)
  • บริหารพื้นที่จัดเก็บ
  • ดูแลความปลอดภัย
  • จัดการการเข้าถึงพร้อมกัน (Concurrency)
  • SQL (Structured Query Language) - ภาษามาตรฐานสำหรับจัดการข้อมูล

  • Information: ประมวลผลลบจำนวนสินค้าที่ขายไป ทำให้รู้ยอด "สินค้าคงเหลือ" ณ ปัจจุบัน

ตัวอย่าง 2: การแยก Data Object▼

หากเราพัฒนาระบบลงทะเบียนเรียน:

  • "นักศึกษา" และ "รายวิชา" ไม่ใช่วัตถุ (Object) เดียวกัน
  • ไม่สามารถนำข้อมูลมาปนในตารางเดียวกันได้
  • ต้องแยก Object ออกมา เช่น เก็บตารางนักศึกษาแยกกับตารางรายวิชา

เช่นเดียวกันกับระบบโรงพยาบาล: "ผู้ป่วย" กับ "อาการโรค (Symptom)" ก็เป็นคนละ Object ที่ต้องแยกกันเก็บ

Exam Notes

เคล็ดลับการเก็งข้อสอบ

1. การวิเคราะห์หน้าจอ (UI Analysis)

ข้อสอบอาจให้ภาพหน้าจอเว็บไซต์มา (เช่น เว็บอัปเดตราคาตลาดหุ้น หรือเว็บจองตั๋ว) แล้วให้วงกลมระบุว่าจุดไหนคือ Data จุดไหนคือ Information

ทริค: ตัวเลขกราฟดิบๆ วันที่ = Data | ยอดรวม ผลต่างเปอร์เซ็นต์ กราฟแท่ง = Information

2. การคำนวณ Storage

อาจมีคำถามสมมติว่าสร้างคอลัมน์ชื่อพนักงานขนาด 10 ตัวอักษร จะกินพื้นที่ฮาร์ดดิสก์กี่ Bits (คำตอบ: 10 × 8 = 80 bits)

3. ระบุ Data Object และ Primary Key

จะมีโจทย์ธุรกิจมาให้ (เช่น ระบบสระว่ายน้ำ) แล้วให้เราหา Object ให้เจอ และระบุ Field ที่ควรเป็น "คีย์หลัก" พร้อมขีดเส้นใต้

สรุปสิ่งที่ต้องจำ (Exam Focus!)

  • Data vs Information: อะไรที่เกิดจากการคำนวณ/ประมวลผล = Information ทั้งสิ้น (อายุ, GPA, BMI, กราฟ)
  • อายุไม่เก็บใน DB: เพราะ Information เป็น Dynamic เปลี่ยนแปลงตลอดเวลา เก็บแค่วันเกิด (Data) แล้วให้โปรแกรมคำนวณเอา
  • Primary Key: ตัวจำแนกเอกลักษณ์ มักขีดเส้นใต้ เช่น รหัสนักศึกษา, ISBN, Barcode
  • Storage: 1 ตัวอักษร = 1 Byte = 8 Bits
  • File System ไม่ดี: ทำให้เกิด Data Anomalies 3 แบบ (เพิ่ม, แก้ไข, ลบ) จาก Data Redundancy

ข้อควรระวัง - นักศึกษามักสับสน

  • Excel ไม่ใช่ Database: เป็นแค่ Computerized file systems
  • Wisdom ไม่บันทึกใน Database: มีแค่ Data (ใน DB) และ Knowledge (ใน Knowledge Base) การตัดสินใจของมนุษย์ไม่เก็บในคอมพิวเตอร์
  • Attribute ไม่ใช่ Object: "อาการไอ" ไม่ใช่คอลัมน์ของผู้ป่วย แต่เป็น Object ใหม่ที่ต้องแยกเป็นตารางต่างหาก
