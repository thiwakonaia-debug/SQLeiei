TOPIC 02

DIKW Hierarchy

Data
ข้อมูลดิบ (Raw Fact) — บันทึกลงสื่อ เปลี่ยนแปลงไม่ได้ เช่น วันเกิด · ราคาสินค้า · น้ำหนัก · บาร์โค้ด · วันหมดอายุ
Information
ผ่านการประมวลผล (Dynamic) — ไม่บันทึกใน DB ให้ UI คำนวณ เช่น อายุ · GPA · BMI · ยอดขายรวม · กราฟ · เปอร์เซ็นต์ทุกชนิด
Knowledge
รูปแบบสถิติจาก Information (เก็บเป็น Knowledge Base ได้) เช่น "ลูกค้าซื้อน้ำ → มักซื้อขนมปังด้วย" ระบบแนะนำโปรโมชั่น
Wisdom
การตัดสินใจของมนุษย์ — คอมพิวเตอร์เก็บไม่ได้ เช่น ลูกค้าตัดสินใจ "หยิบ" หรือ "ไม่หยิบ" ขนมปังตามที่ระบบแนะนำ
🎯 ออกสอบแน่ ! อาจารย์จะให้ดูหน้าจอ (ราคาทอง / พยากรณ์อากาศ) แล้วถามว่า "ตรงนี้คือ Data หรือ Information?"
ทริค: เวลา (14:43 น.) = Data · ราคาปรับลด 50 บาท = Information · กราฟ/เปอร์เซ็นต์ = Information ทั้งหมด

TOPIC 03

ปัญหาระบบแฟ้มข้อมูล (File System)

⚠️ File System (Excel/CSV) — ปัญหา
  • Data Redundancy: ข้อมูลซ้ำซ้อน เปลืองพื้นที่ นำไปสู่ Data Inconsistency
  • Deletion Anomaly: ลบข้อมูลหนึ่งแล้วข้อมูลอื่นหายตาม
  • Update Anomaly: อัปเดตข้อมูลเดียวต้องแก้หลายที่
  • Structural Dependence: เปลี่ยนขนาดคอลัมน์ = ต้องรื้อโค้ดทั้งหมด
✅ Database System (DBMS) — ข้อดี
  • ลดความซ้ำซ้อน ข้อมูลเก็บที่เดียว อ้างอิงข้ามตารางได้
  • Data Sharing: ผู้ใช้หลายคนเข้าถึงได้พร้อมกัน
  • Data Independence: เปลี่ยนโครงสร้าง DB ไม่กระทบโปรแกรม
  • Security & Integrity: ควบคุมสิทธิ์และความถูกต้องได้
TOPIC 04

ระบบฐานข้อมูล & Data Object

🗄️ DBMS คืออะไร?

ซอฟต์แวร์ตัวกลาง (Intermediary) ระหว่างผู้ใช้กับฐานข้อมูล เช่น MySQL, Oracle, PostgreSQL, MS SQL Server

🎯 Data Object & Primary Key

การออกแบบ DB ที่ดีต้องแยก Object ให้ถูกต้อง

ตัวอย่างระดับสอบ:
· นักศึกษา ≠ รายวิชา → ต้องแยกตาราง
· คนไข้ ≠ อาการป่วย → ต้องแยกตาราง

Primary Key ตัวอย่าง:
· นักศึกษา = รหัสนักศึกษา · สินค้า = บาร์โค้ด · หนังสือ = ISBN

🎯 ออกสอบแน่ ! อาจารย์จะให้โจทย์ธุรกิจ (เช่น สระว่ายน้ำ, สนามกีฬา) แล้วถามว่ามี Data Object อะไรบ้าง และอะไรคือ Primary Key

สูตร

การคำนวณที่ต้องรู้

สูตรที่ 1 — แยก Data vs Information
Age = Current Year − Birth Year
Birth Year = Data (บันทึกครั้งเดียว) · Age = Information (เปลี่ยนทุกปี ไม่เก็บใน DB)
สูตรที่ 2 — คำนวณพื้นที่จัดเก็บ
1 ตัวอักษร = 1 Byte = 8 Bits
ตัวอย่างสอบ: Field "AKA" ใช้กี่ Bit? → 3 ตัวอักษร × 8 = 24 Bits
🎯 ออกสอบแน่ ! ถามการคำนวณ Bit/Byte ของชื่อหรือคำในฐานข้อมูล เช่น "หากชื่อ Field มี N ตัวอักษร ใช้กี่ Bit?"

💡 ทำไมไม่เก็บ Information ในฐานข้อมูล?

เพราะมันเป็น Dynamic — เปลี่ยนแปลงได้ตลอดตามเวลาและปัจจัยแวดล้อม
ถ้าเก็บ "อายุ" ไว้ใน DB วันพรุ่งนี้ข้อมูลก็ผิดแล้ว

🏢 DIKW + 7-Eleven (ตัวอย่างที่อาจารย์ใช้)

· Data: บาร์โค้ดสินค้า, ราคา, วันหมดอายุ
· Information: จำนวนขวดน้ำที่เหลือ, ยอดขายรวมวันนี้
· Knowledge: "ลูกค้าซื้อน้ำ มักซื้อขนมปัง" → ขึ้นโปรโมชั่นที่แคชเชียร์
· Wisdom: ลูกค้าตัดสินใจหยิบหรือไม่หยิบขนมปัง

อ่านก่อนสอบ ⚡ Cheat Sheet
Data Hierarchy

Bit → Byte (8 bits) → Field → Record → File → Database

Data

Raw Fact บันทึกลงสื่อ เปลี่ยนไม่ได้ เช่น วันเกิด, บาร์โค้ด, น้ำหนัก

Information

ผ่านการคำนวณ Dynamic เช่น อายุ, GPA, ยอดขาย, กราฟ, % ไม่เก็บใน DB

Knowledge

รูปแบบสถิติจาก Information เช่น ซื้อ A → ซื้อ B (Knowledge Base ได้)

Wisdom

การตัดสินใจมนุษย์ คอมพิวเตอร์เก็บไม่ได้

Primary Key

ตัวระบุ Object: รหัสนักศึกษา, บาร์โค้ด, ISBN ต้องไม่ซ้ำ

DBMS

ตัวกลางระหว่างผู้ใช้กับ DB แก้ปัญหา Redundancy & Anomalies

Structural Dependence

ข้อเสีย File System ถ้าเปลี่ยนความยาว Column โปรแกรมพังทั้งหมด

🗺️ Mind Map

📚 ITDS 221 — Database Management Systems
1. DIKW Hierarchy
  • Raw Fact → ความจริงดิบ ติดตัววัตถุ
  • Data → บันทึกลงสื่อ เปลี่ยนไม่ได้ (วันเกิด, บาร์โค้ด)
  • Information → คำนวณแล้ว Dynamic (อายุ, กราฟ) ไม่เก็บใน DB
  • Knowledge → รูปแบบสถิติ (ซื้อ A → ซื้อ B) เก็บเป็น Knowledge Base
  • Wisdom → การตัดสินใจมนุษย์ คอมพิวเตอร์เก็บไม่ได้
2. Data Hierarchy & Storage
  • Bit (0,1) → Byte (8 bits = 1 char) → Field → Record → File → Database
  • 1 ตัวอักษร = 1 Byte = 8 Bits
3. File System vs Database System
  • File System: Redundancy, Anomalies, Structural Dependence
  • DBMS: ลดซ้ำซ้อน, Data Sharing, Data Independence
4. Database Design Concept
  • Data Object: สิ่งที่สนใจ เช่น นักเรียน, คนไข้, สินค้า
  • Primary Key: Identifier ไม่ซ้ำ เช่น รหัสนักศึกษา, ISBN, Barcode
5. Types of Database
  • ตามผู้ใช้: Single, Workgroup, Enterprise
  • ตามสถานที่: Centralized, Distributed, Cloud
  • ตามโครงสร้าง: Structured, Semi-structured, Unstructured
