Data Semantic

ความหมายของข้อมูล

คอมพิวเตอร์ไม่สามารถเข้าใจข้อมูลได้เหมือนมนุษย์ จึงต้องมี Data Dictionary (พจนานุกรมข้อมูล) เพื่ออธิบายความหมาย (Semantic) ของข้อมูลแต่ละตัว

มนุษย์ vs คอมพิวเตอร์

🧠 มนุษย์เข้าใจทันที: เห็นสัญลักษณ์ ♂ และ ♀ → รู้ทันทีว่าเป็นชายและหญิง

🤖 คอมพิวเตอร์ไม่เข้าใจ: เห็นตัวอักษร "F" ในตาราง → ไม่รู้ความหมายว่าคือเพศหญิง

Key Point

ต้องสร้าง Data Dictionary เพื่ออธิบายว่า F = Female ให้ระบบเข้าใจ

สรุป

Data Semantic = ความหมายแฝงของข้อมูล / Data Dictionary = เครื่องมือแปลความหมายให้เครื่อง

4 Basic Building Blocks

องค์ประกอบพื้นฐานของ Data Model

แบบจำลองข้อมูลประกอบด้วย 4 องค์ประกอบหลัก: Entity, Attribute, Relationship, Constraint

Entity

แปลงมาจาก Data Object ในโลกจริง สิ่งที่เราต้องการจัดเก็บข้อมูล

Physical / Tangible

วัตถุที่จับต้องได้ เช่น คน (ลูกค้า, นักเรียน), สถานที่ (ห้องเรียน), ทรัพย์สิน (สินค้า)

Logical / Intangible

สิ่งที่จับต้องไม่ได้: กระบวนการ (Process) หรือ นามธรรม (Abstract) เช่น การลงทะเบียน (Enrollment), การสอบ (Examination), กำไร (Profit), อุบัติเหตุ (Accident)

Attribute

คุณลักษณะของ Entity = Column/Field ในตาราง เช่น ID, Name

Relationship

เชื่อมโยงระหว่าง Entity 2 ตัวขึ้นไป: 1:1, 1:M, M:N

Constraint

ข้อบังคับเสริม (Optional) มักใช้คำว่า "ต้อง (Must)"

ข้อควรระวัง

ข้อสอบมักหลอกด้วยสิ่งที่เป็นนามธรรมหรือกระบวนการ เช่น "การลงทะเบียน" — คำตอบคือ เป็น Logical Entity ได้!

ตัวอย่าง Constraint

"Student can enroll" (แค่ข้อเท็จจริง) vs "Student must enroll 5 subjects" (Constraint — มีเงื่อนไขบังคับ)

สรุป

Entity = คำนาม | Attribute = Schema/Column | Relationship = คำกริยา | Constraint = คำว่า "ต้อง"

Business Rules & Forward/Backward Analysis

การวิเคราะห์ความสัมพันธ์

Business Rule ที่ดี ประกอบด้วย ประธาน + กริยา + กรรม โดยคำนามแปลงเป็น Entity และคำกริยาแปลงเป็น Relationship จากนั้นต้องวิเคราะห์ทั้ง Forward และ Backward เพื่อหาความสัมพันธ์ที่แท้จริง

Forward Analysis (Active Voice)▼

Forward: "Student attend classes" → 1:M (นักเรียน 1 คน เข้าเรียนได้หลายคลาส)

Business Rule
→
Forward (Active Voice)
→
Student attend classes (1:M)
Backward Analysis (Passive Voice)▼

Backward: "Classes are attended by students" → 1:M (คลาส 1 ห้อง ถูกเรียนโดยนักเรียนหลายคน)

คิดย้อนกลับ
→
Backward (Passive Voice)
→
Classes are attended by students (1:M)
สรุป: Forward + Backward = M:N▼

🏆 สรุป: M:N = Forward (1:M) + Backward (1:M)

รวมกัน = Many-to-Many

การแปลง Business Rule

  • คำนาม (Noun) → แปลงเป็น Entity
  • คำกริยา (Verb) → แปลงเป็น Relationship
  • "ต้อง (must)" → แปลงเป็น Constraint

ข้อควรระวัง

ห้ามวิเคราะห์เพียงฝั่งเดียว! ถ้าโจทย์ให้มาขาเดียว ต้องคิดย้อนกลับเองเสมอ

Key Takeaway

Forward (ซ้าย→ขวา, Active) + Backward (ขวา→ซ้าย, Passive) = ความสัมพันธ์ที่แท้จริง

วิวัฒนาการของ Data Models

ประวัติศาสตร์ Data Model

Data Model พัฒนามาตามยุคสมัยตั้งแต่ปี 1960 จนถึงรูปแบบที่ใช้ในปัจจุบัน

ค.ศ. 1960
🌲 Hierarchical Model
โครงสร้างคล้ายต้นไม้กลับหัว (Upside-down tree) สร้างความสัมพันธ์แบบ Parent-Child (1:M)
ค.ศ. 1969
🕸️ Network Model
พัฒนาขึ้นแก้ปัญหา Hierarchical โดยให้ Record หนึ่งมี Parent ได้มากกว่า 1 ตัว แต่มีความซับซ้อนสูง
ค.ศ. 1970
📊 Relational Model
นำเสนอโดย E.F. Codd (IBM) ใช้ตาราง (Table) ประกอบด้วยแถว (Tuple) และคอลัมน์ (Attribute) ใช้คู่กับ RDBMS และ SQL
ค.ศ. 1976
📐 Entity-Relationship Model
นำเสนอโดย Peter Chen ใช้แผนภาพ (ER Diagram) ช่วยให้นักออกแบบมองเห็นภาพรวมได้ง่ายขึ้นก่อนแปลงเป็น Relational Model

Key Takeaway

ลำดับจำง่าย: Hierarchical (1960) → Network (1969) → Relational (1970, Codd) → ER Model (1976, Chen)

ตัวอย่างประกอบ

ตัวอย่างจากอาจารย์

♂♀ ตัวอย่างที่ 1: สัญลักษณ์เพศ (Data Semantics)▼

มนุษย์เข้าใจสัญลักษณ์ ♂ และ ♀ ทันทีว่าเป็นชายหรือหญิง แต่คอมพิวเตอร์ไม่เข้าใจ

ตัวอย่าง

ต้องมี Data Dictionary บอกความหมายว่า "F" ที่เก็บในระบบแปลว่า Female

🌡️ ตัวอย่างที่ 2: อุณหภูมิ กำไร อุบัติเหตุ (Logical/Abstract)▼

สิ่งที่เป็นนามธรรมจับต้องไม่ได้ แต่สามารถแปลงเป็นข้อมูลกายภาพเพื่อจัดเก็บได้

ตัวอย่าง

อุณหภูมิวัดเป็นองศา / อุบัติเหตุบันทึกเป็นสถานที่เกิดเหตุ — ล้วนเป็น Logical Entity ที่เก็บข้อมูลได้

📝 ตัวอย่างที่ 3: can vs must (Data Constraint)▼

"Student can enroll" → เป็นเพียงคุณลักษณะทั่วไป

"Student must enroll" → เป็นการกำหนด Constraint ทันที (If-Then)

ข้อควรระวัง

คำว่า "must" คือสัญญาณบ่งบอก Constraint ในประโยค

📱 ตัวอย่างที่ 4: สมาร์ทโฟน/อีเมล/ลงทะเบียน (3 Types of Relationship)▼
  • นักเรียน 1 คน มีมือถือ 1 เครื่อง → 1:1
  • นักเรียน 1 คน มีหลายอีเมล → 1:M
  • นักเรียนหลายคน ลงทะเบียนหลายวิชา → M:N
🎒 ตัวอย่างที่ 5: Student attend classes (Active vs Passive)▼

Forward: "นักเรียนหลายคนเข้าเรียนคลาส" → 1:M

Backward: "คลาสหนึ่งถูกเรียนโดยนักเรียนหลายคน" → 1:M

สรุป

รวมกัน = M:N

จุดที่มักออกสอบ (Midterm)

Exam Traps ที่ต้องระวัง

⚠️ Trap 1: Physical vs Logical Entity

ข้อสอบหลอกด้วยสิ่งนามธรรม/กระบวนการ เช่น "Enrollment", "Examination" → คำตอบคือเป็น Logical Entity ได้ เพราะเป็น Process

⚠️ Trap 2: Forward/Backward Analysis

ให้ประโยคมาทางเดียว แล้วถามความสัมพันธ์จริง → ต้องวิเคราะห์ Passive Voice ย้อนกลับเสมอ เพื่อเช็คว่าเป็น M:N หรือไม่

⚠️ Trap 3: แปลง Business Rule เป็น ERD

ให้ Statement มา แล้วถามว่าคำใดเป็น Entity/Relationship หรือหา Attribute ที่ซ่อนอยู่ → ต้องแยก Noun = Entity, Verb = Relationship

⚠️ Trap 4: Schema vs Instance

ต้องแยกให้ออก: Schema = โครงสร้าง (Column Name) ส่วน Instance = ข้อมูลดิบ (Raw Data) ที่อยู่ข้างใน
